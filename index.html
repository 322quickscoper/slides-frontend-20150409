HTML — абетка веба

Перша літера в ній — doctype

			<!doctype html>
			<meta charset="utf-8">
			<title>Hello World</title>
			<link rel="stylesheet" href="style.css">
			<script src="script.js"></script>

Типова сторінка

			<!doctype html>
			<title>Назва</title>
			<link rel="stylesheet" href="style.css">
			<script src="modernizr.js"></script> ~10KB
			<p>Абзац</p>
			<script src="//ajax.googleapis.com/…/jquery.js"></script>
			<script src="script.js"></script>

HTML5 ★ BOILERPLATE

https://html5boilerplate.com

  • Базова структура html
  • Основний css
  • Системні файли з налаштуваннями: .editorconfig, .gitignore і т.д.
  • Modernizr, jQuery
  • Іконки для браузерів та пристроїв — favicon.ico, apple-touch-icon.png

http://media.1plus1.ua/apple-touch-icon.png

Припустимо, що пройшов деякий час і ми вже щось вміємо і знаємо

А саме:

  1. Знаєм html та семантику його тегів
  2. Вільно стилізуємо навіть сторінки зі складною структурою
  3. Збираємо Modernizr відповідно до потреб конкретної верстки
    http://modernizr.com/download/
  4. Можемо підключити карусель на jQuery
  5. Розуміємо, чому верстальники недолюблюють IE
  6. Знаємо, що значать всі ці букви

Ми готові йти далі до фреймворків

Якщо html — це абетка, то Bootstrap — словник

Bootstrap став незамінним інструментом, що дозволяє швидко розгортати функціональні прототипи та є зрозумілим як для верстальника, так і для програміста. Для нас він став конвенцією написання форм, кнопок та інших інтерфейсних компонентів.
http://getbootstrap.com

ZURB Foundation

Гарний приклад фреймворка, альтернативного bootsrap'у. Дуже схожа компонентна структура, відмінність швидше історична ніж функціональна. За відстуності bootstrap, ми б використовували foundation.
http://foundation.zurb.com

Приклад: комбінаторний "вибух"

			p, a, ul, li { // style.less
			  border-top: 2px dotted #366;
			  & + & {
			    border-top: 0;
			  }
			}

Результат: комбінаторний "вибух"

			p, a, ul, li { /* style.css */
			  border-top: 2px dotted #366;
			}
			p + p, p + a, p + ul, p + li, a + p, a + a, …
			li + a, li + ul, li + li {
			  border-top: 0;
			}

А як же картинки?

Використовуємо node-spritesheet

  • Даємо довільну кількість вхідних зображень
  • Отримуємо один зібраний спрайт
  • + стилі для кожного зображення в .less
  • Є можливість генерації окремих спрайтів для retina-екранів (@2x, @3x)
  • Потребує встановленої бібліотеки ImageMagick :(

Існують бібліотеки-аналоги, які ми теж використовуємо

Постає питання — які пакети на якому проекті використовуються?

Розгадка в package.json

			{
			  "name": "ideil",
			  "devDependencies": {
			    "grunt": "^0.4.4",
			    "node-spritesheet": "^0.4.2"
			  }
			}

Встановлюємо npm-пакети

			$ cd ideil && npm install

Всі пакети та їх залежності потрапляють в папку node_modules, яка зазначена в .gitignore. Таким чином, вони ніколи не потрапляють в репозиторій з кодом, а встановлюються тільки в середовищі розробника.

А як запам'ятати всі команди для збору .less, .js та спрайтів?

Grunt і Gulp — просто менеджери задач

  • Налаштування і опис задач в файлах Gruntfile.js і gulpfile.js
  • Дещо різна ідеологія:
    Grunt — декларативний принцип опису задач
    Gulp — імперативний
  • Відрізняються наявністю доступних пакетів

Gruntfile.js

			grunt.initConfig({
			  less: {
			    compile: { // grunt less:compile
			      files: {'app.min.css': ['app.less']}
			    }
			  }
			});

gulpfile.js

			gulp.task('less', function () {
			  return gulp.src('./less/app.less')
			    .pipe(less())
			    .pipe(gulp.dest('./css'));
			});

І все-таки: що робити з усіма командами?

Описуємо свою задачу, яка буде по черзі виконувати все за нас

			grunt.registerTask('build', [ // grunt build
			  'less',
			  'uglify',
			  'spritesheet'
			]);

Можна автоматично збирати .less, .js та спрайти за допомогою grunt-contrib-watch

			watch: { // $ grunt watch
			  less: {
			    files: ['./less/**/*.less'],
			    tasks: ['less']
			  }
			}
